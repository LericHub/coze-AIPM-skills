## 技能操作步骤

1. 处理用户输入的初始任务需求
2. 根据 7 个核心维度进行深度追问，确保需求清晰
3. 将需求澄清结果以 Markdown 格式输出到clarify.md
4. **将生成的 Markdown 内容直接输出到用户消息中**（如果包含 Mermaid 代码块，需要特殊说明以便渲染）
5. 等待用户确认交付物

---

## 技能描述

新手往往难以一次性说清需求。AI 应首先采用「灵活对话式」引导，主动评估、温和追问、并进行总结确认。在需求未明确前，**不要急于创建文件或文件夹**。

### 1.1 接收需求

- 接受用户的任意形式输入（一句话、截图、草图、大白话）。
- **不要**要求用户填写复杂的固定模板，降低认知门槛。

### 1.2 核心维度评估与反馈

当你接收到用户的初步需求后，**必须将以下 7 个关键维度的评估结果直接展示给用户**，告诉他们当前需求在哪些维度是清晰的，哪些是缺失的：

1. **背景/痛点**：为什么要做？现状有什么问题？（必须）
2. **业务目标**：做完后想达到什么效果？（必须）
3. **上下游系统和系统边界**：项目需要对接哪些上下游？各自系统提供什么数据或能力？（必须）
4. **接口文档或文档地址**：若涉及上下游，请用户提供文档地址（必须）
5. **业务规则**：有哪些必须遵守的限制或特定流程？是否有特殊业务规则需要注意？（重要）
6. **业务风险**：有没有需要注意的业务风险（重要）
7. **埋点场景确认**：确认除记录PV、UV外，是否还有其他埋点场景？（重要）
8. **参考/竞品**：有没有喜欢的对标产品？（加分项）
9. **Not gold**：有什么场景是这次迭代不需要做的？（加分项）

黄金准则：
- 只针对产品层面进行提问，非功能需求不在范围内
- 不需要陷入细节，只需要关注产品层面的需求。细节内容在细化阶段由你补充，用户审核确认。

## 🚨 深度追问的重要规则

### ❌ 绝对禁止询问的问题（技术实现方式）

**禁止询问技术实现方式！这些应该由你来整理，而不是问用户：**

| 禁止询问的问题类型 | 错误示例 |
|-----------------|---------|
| 数据库设计 | "用户表需要哪些字段？" |
| 接口设计 | "这个接口需要什么参数？" |
| 技术架构 | "用什么数据库？用什么框架？" |
| 数据结构 | "订单对象有哪些属性？" |
| 缓存策略 | "需要缓存吗？缓存多久？" |
| 消息队列 | "需要用消息队列吗？" |
| 安全认证 | "用什么认证方式？" |

---

### ✅ 应该询问的问题（业务/功能层面）

**只询问业务和功能问题，引导用户描述他们想要什么：**

| 应该询问的问题类型 | 正确示例 |
|-----------------|---------|
| 业务目标 | "您希望这个功能解决什么业务问题？" |
| 用户场景 | "用户在什么情况下会使用这个功能？" |
| 功能需求 | "用户具体能做什么操作？" |
| 业务规则 | "有什么特殊的业务规则需要遵守吗？" |
| 异常场景 | "如果出现XX情况，应该怎么处理？" |
| 用户期望 | "您希望最终达到什么样的效果？" |
| 参考产品 | "有没有您觉得做得好的参考产品？" |

---

### 📝 追问技巧示例

#### ❌ 错误的问法（询问技术实现）
```
错误：
"用户表需要包含哪些字段？"
"这个接口的入参和出参是什么？"
"订单状态机怎么设计？"
```

#### ✅ 正确的问法（询问业务/功能）
```
正确：
"您希望保存用户的哪些信息？"
"用户下单时需要填写哪些信息？"
"订单有哪些状态变化？"
"在什么情况下订单会被取消？"
```

---

### 🎯 启发式追问策略

**核心思路：通过启发式提问，引导用户全面思考需求，而不是直接询问技术细节。**

#### 策略 1：场景化追问
```
"想象一下，一个用户第一次使用这个功能，他会经历什么样的流程？"
"在这个场景中，用户最关心的是什么？"
```

#### 策略 2：目标导向追问
```
"您希望通过这个功能达到什么具体目标？"
"成功的衡量标准是什么？"
```

#### 策略 3：边界探索追问
```
"有没有什么场景是这次不需要考虑的？"
"这个功能和其他功能的边界在哪里？"
```

#### 策略 4：异常场景追问
```
"如果出现XX情况，您希望怎么处理？"
"最坏的情况是什么？应该如何应对？"
```

## 🚨 追问自检清单

**每次追问前，必须逐项检查：**

- [ ] 这个问题是询问业务/功能层面的吗？
- [ ] 这个问题不是询问技术实现方式吗？
- [ ] 这个问题能引导用户深入思考需求吗？
- [ ] 这个问题是开放式的，不是封闭式的吗？
- [ ] 这个问题不会让用户觉得在做技术设计吗？

**只有所有检查项都通过，才能提出这个问题！**

---

## 1.3 深度追问与多轮确认 【等待用户确认】

**🚨 关键规则：不要急于总结，必须进行至少 1 轮深度的启发式追问！**
- **追问方式**：抛弃死板的选项（A/B/C），采用**开放式、启发式**的提问，引导用户多方面、更全面地思考这个需求。
- **追问数量**：每次追问必须在 **1 到 3 个问题**之间。
- **循环机制**：基于用户的上一次回答，挖掘出新的盲点继续追问，至少进行 1 轮对话，最多进行3轮对话。

**停止规则（二选一）：**
- **条件A**：LLM 在三轮提问中认为业务背景已经清晰，可以开始结构化分析需求
- **条件B**：当前已完成的提问轮次达到三轮

满足任一条件即停止追问，进入最终确认环节。

- **最终确认**：只有当你通过多轮追问**完全理解**了需求的所有细节，并且用户明确表示没有补充了，可以开始写了之后，你才能进行结构化复述并进入下一步。
- **🛑 强制暂停：在每一轮提问后必须停止输出！等待用户回复后才能进行下一轮动作。**

7. 确认后 → 调用 manage.persist_memory()
