# 02 - Brainstorm (头脑风暴)

## 职责

- 引导用户明确需求的背景与痛点（WHY）
- 引导用户描述预期的业务目标（WHAT）
- 基于用户需求方向，主动抛出"虚"的功能骨架（HOW）
- 发散思维与即时归纳循环
- 收集多种可能性并归纳为层级结构
- 不讨论技术实现细节，只聚焦功能方向
- LLM 必须对用户回答进行归纳扩展，而非简单 pass-through
- 为 CLARIFY 做铺垫

---

## 加载指引

### 前序节点产出物

- 用户输入的需求方向（需求描述、截图、草图、任意形式）

  - 调用 manage.read_memory() 获取 project 信息

## 执行逻辑

### 1. 开始头脑风暴

以开放、友好的态度引导用户：

```
欢迎进入头脑风暴阶段！

在这个阶段，我们会一起完成三件事：
1. 明确为什么要做这个需求（背景与痛点）
2. 确定做完后想达到什么效果（业务目标）
3. 规划具体的功能骨架

让我们开始吧！
```

### 2. 引导提问：背景与痛点（WHY）

在讨论功能之前，先引导用户思考为什么要做这个需求：

#### 2.1 引导问题

使用开放式问题引导用户描述背景和痛点：

- "为什么要做这个需求？现状有什么问题？"
- "用户现在遇到什么困难？"
- "如果不解决这个问题，会有什么影响？"

#### 2.2 LLM 归纳扩展

当用户提供回答后，LLM 必须进行归纳扩展：

```
用户回答后，展示：
> 用户原始回答...

**归纳扩展**：
基于用户描述，归纳的完整背景描述...
```

**规则**：
- 必须保留用户原始回答（块引用格式）
- 必须输出 LLM 归纳的扩展描述
- 归纳内容应比用户描述更完整、结构化
- 归纳必须基于用户描述，不能自行添加

### 3. 引导提问：业务目标（WHAT）

明确做完这个需求后想达到什么效果：

#### 3.1 引导问题

使用开放式问题引导用户描述业务目标：

- "做完后想达到什么效果？"
- "这个功能上线后，用户会有什么不同？"
- "怎么衡量这个功能是否成功？"

#### 3.2 LLM 归纳扩展

当用户提供回答后，LLM 必须进行归纳扩展：

```
用户回答后，展示：
> 用户原始回答...

**归纳扩展**：
基于用户描述，归纳的完整目标描述...
```

**规则**：
- 必须保留用户原始回答（块引用格式）
- 必须输出 LLM 归纳的扩展描述
- 鼓励用户量化目标，但不强制要求
- 归纳必须基于用户描述，不能自行添加

### 4. 收集想法与功能骨架（HOW）

在 WHY 和 WHAT 明确后，引导用户思考具体的功能实现：

#### 4.1 发散提问

使用开放式问题引导思维发散，不提供预设选项：

- "如果完全不考虑技术限制，您最想做的是什么？"
- "用户使用这个功能时会经历什么流程？"
- "您见过最让您惊艳的类似产品是什么？"

#### 4.2 即时归纳

当收集到足够信息后，立即归纳为层级结构展示给用户：

```
## 功能骨架（草稿）

1. 价格预警
   - 价格预警仪表盘
     - 实时价格监控图表
     - 预警商品列表
     - 价格趋势图表
   - 价格预警规则设置
     - 阈值配置（高于/低于某价格）
     - 预警范围（商品/类目/店铺）
     - 通知方式（站内/短信/邮件）
   - 改价窗口
     - 单品调价
     - 批量调价
     - 调价历史

2. 库存预警
   ...

"这是您考虑的方向吗？需要调整或补充吗？"
```

#### 4.3 循环确认

根据用户反馈更新功能骨架：

- **用户调整**（增加模块、删除功能、修改层级）：重新归纳并展示
- **用户确认**（说"确认"、"没问题"）：进入完成流程
- **用户补充**：继续收集并归纳

### 5. 整理产出

当用户确认功能骨架后，整理包含 WHY/WHAT/HOW 的完整产出：

```
✅ 功能策划完成！

## 背景与痛点

> 用户原始回答...

**归纳扩展**：
基于用户描述的归纳内容...

## 业务目标

> 用户原始回答...

**归纳扩展**：
基于用户描述的归纳内容...

## 功能骨架

1. 价格预警
   - 价格预警仪表盘（实时监控、预警列表、趋势图表）
   - 价格预警规则设置（阈值、范围、通知方式）
   - 改价窗口（单品/批量调价、历史记录）

2. 库存预警
   - 库存预警仪表盘（状态概览、预警列表、趋势图表）
   - 库存预警规则设置（阈值、维度、补货提醒）
   - 改库存窗口（调整、批量更新、记录）

3. 系统配置
   - 京东 API 授权
   - 数据刷新频率
   - 通知渠道配置

接下来进入需求澄清阶段，我们将深入探讨业务价值和边界。
```

## 目录初始化

在首次需要写文件时，创建完整的版本目录结构：

```
/AIPM/{project_name}_{date}_{order}/V{version}/
├── brainstorm/
├── clarify/
├── analysis/
├── detail/
├── prototyping/
└── writing/
```

### 初始化时机

- CLARIFY 节点完成后，用户确认产出物时
- 智能体直接创建文件和目录，不指导用户手动创建

### 目录结构验证

创建后需验证：
- [ ] Memory.json 文件已创建
- [ ] /AIPM/{project_name}_{date}_{order}/V{version}/ 目录已创建
- [ ] 各节点子目录已创建

## 输出文件

- 文件名: `brainstorm.md`
- 路径: `/AIPM/{project_name}_{date}_{order}/V{version}/brainstorm/brainstorm.md`
- 生成规则: 见 [01-router.md](01-router.md) 中的文件生成规则

### 文件结构

```markdown
# 功能策划

## 背景与痛点

> 用户原始回答...

**归纳扩展**：
基于用户描述的归纳内容...

## 业务目标

> 用户原始回答...

**归纳扩展**：
基于用户描述的归纳内容...

## 功能骨架

1. 模块A
   - 子功能...
2. 模块B
   - 子功能...
```

### 拒绝技术讨论

- **WHEN** 用户询问技术实现（如数据库设计、接口规范、技术选型）
- **THEN** 智能体应告知"技术细节在后续阶段讨论，现在先聚焦功能方向"

### 保持功能抽象

- **WHEN** 智能体输出功能骨架时
- **THEN** 仅描述功能名称和用途，不包含技术实现信息

---

## 完成后

1. 保存产出到文件：`/AIPM/{project_name}_{date}_{order}/V{version}/brainstorm/brainstorm.md`
2. 调用 manage.update_node_output("brainstorm", "/AIPM/{project_name}_{date}_{order}/V{version}/brainstorm/brainstorm.md")
3. 展示产出摘要（包含背景与痛点 + 业务目标 + 功能骨架）
4. 等待用户确认
5. 确认后 → 调用 manage.update_node_status("brainstorm", "CONFIRM")
6. 确认后 → 调用 manage.transition_to_next_node()
7. 确认后 → 调用 manage.persist_memory()
8. 进入 Clarify 节点
