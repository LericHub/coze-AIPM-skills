## Context

当前 product_manager 节点的 "完成后" 章节均提及"更新状态（见各节点内状态更新规范）"，但该规范实际分散在各节点内部或存在于 STATE_RULES.md，未形成统一的管理模块。

现状：
- 各节点产出后需更新节点状态，但更新逻辑不统一
- 记忆持久化（上下文快照）写入时机不明确
- 版本号管理缺少集中入口

## Goals / Non-Goals

**Goals:**
- 统一节点状态管理（DRAFT/CONFIRM 两态）
- 统一记忆持久化逻辑（快照写入/读取）
- 统一版本号管理入口
- 各节点通过引用 manage 而非内嵌逻辑

**Non-Goals:**
- 不改变现有节点的业务职责（clarify 分析、detail 设计等）
- 不改变整体流程（01→02→...→09）
- 不处理文件系统的具体读写操作（由各节点自行执行）
- 记忆和版本号的详细逻辑后续迭代

## Decisions

### 1. manage 定位为共享能力，非线性节点

**决定**: 创建 `_manage.md`（下划线前缀表示内部使用），各节点按需引用。

**理由**: 
- 避免破坏线性流程
- 高频操作需要统一入口
- 下划线命名在目录中排序靠前，便于识别为基础设施

### 2. 节点状态机设计

**决定**: DRAFT/CONFIRM 两态

**理由**: 
- 足够简单，满足需求
- DRAFT：默认、产出后、修改后
- CONFIRM：用户确认后

**状态流转**:
```
用户确认 → DRAFT → CONFIRM → 下一节点
用户修改 → CONFIRM → DRAFT（重新生成）
```

### 3. Memory.md 结构

**决定**: node_list 记录所有9个节点状态，current_node 记录当前位置

**理由**:
- 需求变更时需重走前序节点，依靠完整 node_list 路由
- 完成后回到变更触发的节点

### 4. manage 接口设计

**决定**: 定义以下接口供各节点调用

```markdown
## manage 提供的能力

### 状态管理
- `get_current_node()`: 获取当前节点
- `update_node_status(node_name, status)`: 更新节点状态
- `transition_to_next_node()`: 流转到下一节点

### 记忆管理
- `write_snapshot(node_name, content)`: 写入上下文快照
- `read_snapshot(node_name)`: 读取上下文快照

### 版本管理
- `get_current_version()`: 获取当前版本
- `increment_version()`: 递增版本
```

### 5. 需求变更路由

**决定**: 需求变更时，前序所有节点状态改为 DRAFT，从第一个 DRAFT 节点开始重走，完成后回到变更触发的节点。

**理由**: 
- 保证变更逻辑完整覆盖所有受影响节点
- 回到触发节点确保用户在正确位置继续

## Risks / Trade-offs

- **风险**: manage 的调用时机依赖各节点遵守 "完成后" 规范  
  **缓解**: 在 SKILL.md 中强调调用规范，测试验证

- **风险**: 状态更新可能遗漏（如异常中断）  
  **缓解**: 每次状态变更后持久化 Memory.md

- **风险**: 两态可能不够区分"首次进入"和"修改重跑"  
  **暂不处理**: 首次/重跑由节点内部判断，manage 只需更新状态值
