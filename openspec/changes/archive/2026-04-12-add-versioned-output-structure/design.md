## Context

当前 product_manager 的节点结构采用线性执行模式，每个节点按顺序执行并产出文件。现存在以下问题：

1. **节点顺序问题**：init 节点（06）独立存在但职责与 router 重叠——router 已经判断场景，init 只是在中间执行一个静默操作，逻辑冗余

2. **输出目录问题**：当前目录结构为 `output/V1.0/`，未考虑需求变更场景
   - 需求变更时，前序节点内容需要更新
   - 新版本需要独立的完整目录，历史版本需要保留
   - 当前结构无法满足版本隔离和完整追溯的需求

3. **节点输出不明确**：各节点没有明确说明输出文件名和路径，LLM 可能不按规范输出

**约束**：
- LLM 不一定严格遵循指定的目录结构，需要设计足够灵活的机制
- 需求变更场景下，需要保留完整的历史迭代过程

## Goals / Non-Goals

**Goals:**
- 移除冗余的 init 节点，简化节点结构
- 设计支持版本隔离的输出目录结构，需求变更时保留完整历史
- 明确每个节点的输出文件规范，引导 LLM 正确输出

**Non-Goals:**
- 不修改状态管理逻辑（_manage 能力）
- 不修改节点执行逻辑（各节点的职责不变）
- 不修改需求变更的流转逻辑

## Decisions

### Decision 1: 移除 init 节点，目录初始化合并到 brainstorm

**选择**：目录初始化逻辑合并到 brainstorm 节点（第一个需要写文件的节点）

**理由**：
- router 已判断场景（首次产出 / 当前节点修改 / 需求变更），不需要独立的 init 节点来执行初始化
- brainstorm 是第一个需要创建文件的节点，在此之前创建目录结构最自然
- 简化节点数量，降低复杂度

**替代方案考虑**：
- 在 router 中初始化：router 职责是路由和说明，不应该包含目录创建逻辑
- 在 clarify 中初始化：clarify 已是第二个节点，中间可能有空白期

### Decision 2: 版本隔离的目录结构

**选择**：
```
/项目名_YYYYMMDD_HHMMSS/
├── Memory.md
├── draft/
└── output/
    ├── V1.0/
    │   ├── brainstorm/
    │   ├── clarify/
    │   ├── analysis/
    │   ├── detail/
    │   ├── prototyping/
    │   └── writing/
    └── V2.0/
        ├── brainstorm/
        ├── ...
```

**理由**：
- 每个版本有完整的独立目录，历史版本完整保留
- 版本号递增时创建新目录，不影响历史
- 按节点名划分子目录，结构清晰

**替代方案考虑**：
- 按文件类型分组（doc/、html/）：不适合，节点是更自然的分组维度
- 扁平结构（V1.0_brainstorm.md）：版本迭代时文件重命名复杂

### Decision 3: 每个节点增加输出文件说明

**选择**：在每个节点文件中增加 "输出文件" 章节

**理由**：
- 显式告诉 LLM 应该输出什么文件、在哪个路径
- 引用 router 中的文件生成规则，避免重复说明
- 节点文件本身就是 LLM 的执行指南

**格式**：
```markdown
## 输出文件

- 文件名: `brainstorm.md`
- 路径: `./output/V{version}/brainstorm/brainstorm.md`
- 生成规则: 见 [01-router.md](01-router.md) 中的文件生成规则
```

### Decision 4: 文件生成规则在 router 中统一说明

**选择**：三种场景在 router 中定义

| 场景 | 操作 |
|------|------|
| 首次产出 | 生成新文件 |
| 当前节点修改 | 更新源文件，版本号不变 |
| 需求变更 + 前序内容修改 | 生成新文件 + 版本号+1 |

**理由**：
- 规则统一在一处，避免多处定义导致不一致
- router 是入口，在此说明最完整

## Risks / Trade-offs

### Risk 1: LLM 可能不按指定结构输出文件

**→ Mitigation**: 
- 在每个节点中明确输出路径
- 在 router 中强调目录结构的重要性
- 可考虑在 _manage 能力中增加目录结构校验

### Risk 2: 版本号管理复杂性

**→ Mitigation**: 
- 版本号由 _manage 能力统一管理
- 需求变更触发时由 manage 能力执行版本号递增

### Risk 3: 删除 init 节点后，现有引用需要更新

**→ Mitigation**: 
- 更新 README.md 中的节点概览表
- 更新 _manage.md 中对 init 的引用
- 确保所有节点文件序号连续
