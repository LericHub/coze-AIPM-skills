## ADDED Requirements

### Requirement: 头脑风暴功能策划

在 Brainstorm 节点执行时，智能体应基于用户输入的需求方向，主动抛出"虚"的功能骨架，并引导用户确认或调整。

#### Scenario: 发散提问

- **WHEN** 用户进入 Brainstorm 节点并输入需求方向
- **THEN** 智能体使用开放式问题引导思维发散，不提供预设选项

#### Scenario: 即时归纳

- **WHEN** 智能体收集到足够信息后
- **THEN** 立即归纳为层级结构展示给用户，格式为：一级模块 → 页面/二级 → 子功能

#### Scenario: 功能骨架展示

- **WHEN** 智能体完成归纳后
- **THEN** 展示结构化功能策划，并询问"这是你考虑的方向吗？需要调整或补充吗？"

#### Scenario: 用户确认方向

- **WHEN** 用户确认功能骨架并说"确认"
- **THEN** 智能体保存功能策划结果，进入 Clarify 节点

#### Scenario: 用户调整方向

- **WHEN** 用户提出调整（如增加模块、删除功能、修改层级）
- **THEN** 智能体根据反馈更新功能骨架，循环直到用户确认

### Requirement: 功能层级结构

智能体产出的功能策划必须遵循以下层级结构：

#### Scenario: 一级模块定义

- **WHEN** 归纳功能骨架时
- **THEN** 一级应为抽象功能模块（如：用户管理、订单管理、价格预警、库存预警）

#### Scenario: 页面/二级功能定义

- **WHEN** 归纳功能骨架时
- **THEN** 二级应为页面级功能或子模块（如：用户列表页、用户详情页、价格预警仪表盘）

#### Scenario: 子功能定义

- **WHEN** 归纳功能骨架时
- **THEN** 子功能应为接口级操作（如：添加用户、编辑用户、删除用户、查看详情）

### Requirement: 非技术原则

Brainstorm 节点不讨论技术实现细节，专注于功能方向。

#### Scenario: 拒绝技术讨论

- **WHEN** 用户询问技术实现（如数据库设计、接口规范、技术选型）
- **THEN** 智能体应告知"技术细节在后续阶段讨论，现在先聚焦功能方向"

#### Scenario: 保持功能抽象

- **WHEN** 智能体输出功能骨架时
- **THEN** 仅描述功能名称和用途，不包含技术实现信息
