## Node Specifications

### 01 - router.md (入口路由器)

**职责：**
- 定义全局执行规范
- 提供节点概览表
- 引导 LLM 加载技能文件

**核心内容：**
```markdown
## 执行规范
1. 节点线性执行: router → brainstorm → clarify → analysis → detail → init → prototyping → writing
2. 用户确认: 除 init 外，所有节点完成后需确认
3. 状态管理: 各节点内部处理状态更新（待后续抽取为 manage）

## 节点概览
| 序号 | 节点 | 用户确认 | 说明 |
|------|------|---------|------|
| 01 | router | - | 入口，无状态管理 |
| 02 | brainstorm | ✓ | 头脑风暴 |
| 03 | clarify | ✓ | 需求澄清(7维度) |
| 04 | analysis | ✓ | 需求分析(PRD骨架) |
| 05 | detail | ✓ | 详细设计(流程图) |
| 06 | init | ✗ | 项目初始化（自动执行） |
| 07 | prototyping | ✓ | 原型制作 |
| 08 | writing | ✓ | PRD撰写 |
| 09 | change | ✓ | 变更分析(可选) |

## 节点详情（简短引用）
- brainstorm: see ./02-brainstorm.md
- clarify: see ./03-clarify.md
- ...
```

### 02 - brainstorm.md (头脑风暴)

**职责：**
- 自由发散思维
- 收集多种可能性
- 不做评判，只记录
- 为 CLARIFY 做铺垫

**来源：** 新增节点

### 03 - clarify.md (需求澄清)

**职责：** 从 7 个维度深度挖掘需求

**来源：** 从 `step1_clarify.md` 迁移，保留：
- 7 维度评估框架
- 深度追问策略
- 禁止询问技术实现规则
- 启发式追问技巧

**移除：**
- 初始化相关逻辑（移至 init 节点）

**完成后：**
```markdown
## 完成后
1. 展示产出摘要
2. 等待用户确认
3. 确认后 → 更新状态（见各节点内状态更新规范）
4. 进入下一节点
```

### 06 - init.md (项目初始化)

**职责：** 创建目录结构，初始化 Memory.md

**来源：** 从 `initialization_flow.md` 和 CLARIFY 后初始化逻辑提取

**核心内容：**
```markdown

1. 询问项目名称
2. 生成版本目录名: {项目名}_{YYYYMMDD}_{HHMMSS}
3. 创建目录结构
4. 创建 Memory.md
5. 设置初始状态

## 完成后
- 无需用户确认
- 直接 → 更新状态
- 进入下一节点
```

## 后续迭代：manage 节点设计

### 待设计内容

manage 节点将在后续迭代中单独实现，核心设计方向：

**职责：**
- 统一处理所有状态更新
- 统一的快照写入逻辑
- 统一的对话日志追加

**调用时机：**
```markdown
## 预期调用时机
在以下节点完成后自动调用（无需用户确认）：
- brainstorm ✓ → manage
- clarify ✓ → manage
- analysis ✓ → manage
- detail ✓ → manage
- init → manage
- prototyping ✓ → manage
- writing ✓ → manage
✓ = 用户已确认
```

**预期执行逻辑：**
```markdown
## 预期执行逻辑
1. 读取当前 Memory.md
2. 更新 node_list（标记当前节点完成）
3. 更新 current_node（指向下一节点）
4. 写入快照标签（CURRENT_SNAPSHOT_XXX）
5. 追加对话日志
6. 持久化 Memory.md

## 异常处理
- Memory.md 不存在 → 创建初始结构
- 写入失败 → 重试 3 次，报错
```

## File Naming Convention

节点文件命名：`{序号}-{node_name}.md`

示例：
- `01-router.md`
- `02-brainstorm.md`
- `03-clarify.md`
- ...
- `09-change.md`
