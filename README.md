# 产品经理全流程工作流技能 (Coze PM Skills)

## 项目简介

这是一个基于 Coze 平台构建的产品经理全流程自动化工具，旨在帮助用户将模糊、不完整的需求逐步转化为结构化、可执行的产品需求文档（PRD）。该工具覆盖从需求澄清到PRD撰写的完整流程，确保每个环节都有清晰的产出物。

## 核心特性

✅ **5节点标准流程**：需求澄清 → 需求分析 → 详细设计 → PRD撰写 → 变更分析  
✅ **独立变更分析节点**：增量变更无需整体回退  
✅ **三级用户确认机制**：确保输出质量  
✅ **完整状态持久化**：支持断点恢复  
✅ **历史版本永久保留**：支持版本回溯  

## 系统架构

### 核心组件
- **主控制器**: [SKILL.md](./SKILL.md) - 协调整个工作流程
- **需求澄清节点**: [step1_clarify.md](./PM_skills/step1_clarify.md) - 明确用户初始需求
- **需求分析节点**: [step2_analysis.md](./PM_skills/step2_analysis.md) - 构建初步PRD框架
- **详细设计节点**: [step3-detail_design.md](./PM_skills/step3-detail_design.md) - 细化交互和数据逻辑
- **PRD撰写节点**: [step4_prd_writing.md](./PM_skills/step4_prd_writing.md) - 整合并输出最终PRD文档
- **变更分析节点**: [step5_change_analysis.md](./PM_skills/step5_change_analysis.md) - 处理需求变更

## 工作流程

### 初始化流程
```
用户触发请求
    ↓
检测是否存在Memory.md状态文件
    ↓
NO → 意图识别 + 方案确认 → 用户确认 → 初始化环境
    ↓
YES → 恢复之前的状态 → 定位当前执行节点
```

### 主执行循环
```
WHILE 未到达结束节点:
    1. 调用对应子Skill执行当前节点
    2. 输出当前节点的产出物
    3. 暂停等待用户反馈
    4. 根据用户反馈决定下一步:
       - 确认 → 标记节点完成 → 进入下一节点
       - 修改 → 重新执行当前节点
       - 变更 → 调用变更分析节点
```

## 功能模块详解

### 1. 需求澄清节点 (CLARIFY)
**触发条件**: 任务进入CLARIFY节点

**核心功能**:
- 接收用户任何形式的初步需求输入
- 评估需求在7个关键维度的完整性:
  1. 背景/痛点 (必须)
  2. 业务目标 (必须)
  3. 上下游系统和系统边界 (必须)
  4. 业务规则 (重要)
  5. 业务风险 (重要)
  6. 埋点场景确认 (重要)
  7. 参考/竞品 (加分项)
- 进行多轮启发式追问，挖掘需求细节
- 生成结构化需求描述

**输出**: 结构化的需求描述，包含7个维度的完整信息

### 2. 需求分析节点 (ANALYSIS)
**触发条件**: 任务进入ANALYSIS节点

**核心功能**:
- 基于已确认的需求信息生成初步PRD文档
- 识别用户类型、业务对象、页面列表和埋点方案
- 构建PRD文档结构骨架
- 生成详细的业务流程和页面流程

**输出**: 包含以下内容的PRD初稿:
- 项目基本信息
- 需求背景与目标 (含四列表格)
- 用户类型和端口识别
- 业务对象列表
- 页面流程和页面列表
- 埋点方案
- 异常场景处理

### 3. 详细设计节点 (DETAIL)
**触发条件**: 任务进入DETAIL节点

**核心功能**:
- 细化业务流程、页面流程、数据点位和埋点方案
- 使用Mermaid绘制可视化流程图
- 梳理每个页面的数据点位，说明数据源、处理逻辑、数据输出
- 为每个事件绘制业务流程图、页面流程图和数据流程图

**输出**: 详细设计文档，包含:
- 高层级业务流程图和页面流程图
- 分端口的页面和功能列表
- 按页面维度的细分事件列表
- 按事件输出的细化需求内容
- 详细的数据处理流程
- 业务场景异常处理方案
- 埋点方案

### 4. PRD撰写节点 (WRITING)
**触发条件**: 任务进入WRITING节点

**核心功能**:
- 整合前三步的所有成果
- 输出结构化的最终版产品需求文档

**输出**: 完整PRD文档，包含以下章节:
1. 项目信息与版本记录
2. 需求背景 (现状问题、为什么现在做)
3. 需求目标 (目标类型、描述、衡量指标、目标值)
4. 需求功能清单 (分端口、分模块输出页面列表和功能描述)
5. 用户场景和业务流程 (包含页面列表和业务流程说明)
6. 数据追踪与埋点 (如有)
7. 异常与边界处理 (断网、空状态、无权限等)
8. 页面设计 (以端口-页面维度输出页面详细信息架构)
9. 附件 (数据字典)

### 5. 变更分析节点 (CHANGE_ANALYSIS)
**触发条件**: 用户提出需求变更

**核心功能**:
- 对已进入开发阶段的需求进行变更影响评估
- 输出页面/事件粒度的增量变更方案
- 避免整体回退，仅更新受影响部分

**输出**: 需求变更分析报告，包含:
- 变更前后业务流程图对比
- 变更前后页面流程图对比
- 受影响页面列表
- 受Affected事件列表
- 每个受影响事件的详细变更说明
- 变更后的数据处理逻辑
- 新增/修改/删除的业务场景
- 变更后的埋点方案调整

## 使用场景

### 标准场景
**适用情况**: 用户有初步想法但需求描述不明确
**流程**: CLARIFY → ANALYSIS → DETAIL → WRITING

### 继续执行场景
**适用情况**: 用户需要继续未完成的任务
**流程**: 恢复Memory.md状态 → 定位当前节点 → 从断点继续执行

### 需求变更场景
**适用情况**: 在任意节点用户提出需求变更
**流程**: 当前节点 → CHANGE → 用户确认变更方案 → 回退到受影响节点 → 重新执行受影响部分 → 继续至终点

### 验证与确认场景
**适用情况**: 用户对某个阶段的产出不满意
**流程**: 当前节点 → 用户反馈 → 修改则重执行当前节点 → 确认则进入下一节点

## 技术特点

### 状态管理与持久化
- **Memory.md 文件**:
  - 记录当前执行节点
  - 保存各阶段的产出物
  - 维护用户确认状态
  - 记录变更历史

- **版本控制**:
  - 保留历史版本
  - 支持回滚到特定版本
  - 记录每次变更的影响范围

### 上下文管理规范
- **快照标签机制**: 每个节点使用标准接口进行上下文读写
  - 读取: `CONTEXT_READ: [SNAPSHOT_TAG_NAME]`
  - 写入: `CONTEXT_WRITE: [SNAPSHOT_TAG_NAME]`
- **快照标签流转**:
  - `CURRENT_SNAPSHOT_CLARIFY`: 澄清阶段输出 (Step 1)
  - `CURRENT_SNAPSHOT_ANALYSIS`: 分析阶段输出 (Step 2, 读取 CLARIFY)
  - `CURRENT_SNAPSHOT_DETAIL`: 详细设计阶段输出 (Step 3, 读取 ANALYSIS)
  - `CURRENT_SNAPSHOT_WRITING`: PRD写作阶段输出 (Step 4, 读取 DETAIL)
  - `CURRENT_SNAPSHOT_CHANGE`: 变更分析阶段输出 (Step 5, 读取 WRITING)

### 用户交互机制
- **确认机制**: 在每个节点完成后，都需要用户确认才能进入下一个节点，确保输出质量。
- **修改机制**: 如果用户对某个节点的输出不满意，可以选择修改，重新执行当前节点。
- **变更机制**: 如果用户提出需求变更，系统会调用变更分析节点，评估影响并制定变更方案。

### 错误处理与恢复
- 每个节点执行后会持久化状态
- 系统崩溃后可以从最后状态恢复
- 变更只影响受影响的节点，保留已确认部分

## 快速开始

1. **安装依赖**: 确保已正确安装 Coze 平台
2. **部署技能**: 将整个项目文件夹上传至 Coze 平台
3. **启动流程**: 触发技能开始需求分析流程
4. **按提示操作**: 根据系统提示逐步完成需求澄清、分析、设计和PRD撰写
